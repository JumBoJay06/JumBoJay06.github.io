Kafka 訊息系統、整合多種資料來源之實作

實作出 youbike 的站點異動偵測跟異動訊息消費與通知吧

設定檔

新增 Kafka 的設置

Kafka

這邊不會直接設置 Kafka 的 sevice，讓 Quarkus 自動設置，偷懶一下。
這邊只設置 Kafka 的通訊設置。

  • mp.messaging.outgoing.station-changes-out:
    • 用途: 設定一個名為 station-changes-out 的傳出 Channel。
    • connector: smallrye-kafka: 指定這個 Channel 使用 Kafka 連接器。
    • topic: youbike.station.changes: 告訴連接器，所有發送到這個 Channel 的訊息，都應該被送到 Kafka 中名為 youbike.station.changes 的 Topic。
    • 對應程式碼: 這段設定對應到 YouBikeChangeDetector 中 @Channel("station-changes-out") Emitter<StationChangeEvent> changeEmitter; 這行程式碼。Quarkus 看到 @Channel("station-changes-out")，就會去 application.yaml 找對應的設定，並建立一個符合設定的生產者 (Emitter) 讓我們注入使用。
  • mp.messaging.incoming.station-changes-in:
    • 用途: 設定一個名為 station-changes-in 的傳入 Channel。
    • connector: smallrye-kafka: 同樣使用 Kafka 連接器。
    • topic: youbike.station.changes: 指定這個 Channel 要去監聽 Kafka 中名為 youbike.station.changes 的 Topic。
    • group.id: youbike-change-processor: 設定消費者群組 ID。這很重要，因為同一個群組內只有一個消費者實例會收到訊息，可以避免重複消費。
    • 對應程式碼: 這段設定對應到 ChangeLogProcessor 中 @Incoming("station-changes-in") 這個註解。Quarkus 看到這個註解，就會去 application.yaml 找對應的設定，並建立一個消費者去監聽指定的 Topic，一旦有新訊息就自動呼叫被註解的方法。

Entity

建立要存到 MySQL 的異動 Log，一樣透過PanacheEntity實作與 MySQL 的介面。

DTO

新增一個 DTO，用來 kafka 間傳遞，目的是要跟資料庫隔離。

Mapper

有 Entity 跟 DTO 了，當然也要有 Mapper 來實作轉換，一樣透過mapstruct實作。

YouBikeChangeDetector

目標是定期檢查 YouBike 站點的車輛數是否有變化，若有，則發送一個事件到 Kafka。

  • 排程執行：
    • 我們使用 Quarkus 的 @Scheduled 註解，讓 detectChanges 方法每隔 60 秒自動執行一次。delayed = "10s" 則是讓應用程式啟動後延遲 10 秒再開始第一次任務，確保其他服務都已準備就緒。
  • 資料獲取與比對：
    • 首先，從我們自己的 MongoDB 中撈出所有已儲存的站點資料，並轉換成一個以 stationNo 為鍵、YouBikeStationEntity 為值的 Map，方便快速查找。
    • 接著，透過 @RestClient 注入的 YouBikeApiClient 從 YouBike 官方 API 獲取最新的站點資料。
    • 遍歷最新的資料，用每一筆資料的 stationNo 去 Map 中查找舊資料。如果找到了，就比對 availableBikes (可借車輛數) 欄位。
  • 發現異動並發送事件：
    如果 availableBikes 的數量不一致，就代表發生了異動。
    • 計算變動量 delta。
    • 建立一個 StationChangeEvent DTO 物件，裡面包含了站點編號、名稱、變動前後的數量、變動量以及變動時間。
    • 透過 @Channel("station-changes-out") 注入的 Emitter，將這個事件物件發送到 Kafka。
  • 更新本地資料：
    比對完成後，將這次從 API 獲取到的最新資料，透過 YouBikeStationEntity.persistOrUpdate(youBikeList) 全數更新回 MongoDB，作為下一次排程執行的比對基準。
完整檔案

ChangeLogProcessor

目標是接收從 Kafka 來的異動訊息，並將其儲存到 MySQL 和 Redis。

  • 監聽 Kafka Topic：
    • 我們使用 @Incoming("station-changes-in") 註解來告訴 Quarkus，processChange 這個方法要去監聽名為 station-changes-in 的 Kafka Channel (這個 Channel 在 application.yaml 中被設定為對應 youbike.station.changes topic)。
  • 儲存至 MySQL：
    • 當收到新的 StationChangeEvent 事件時，processChange 方法會被觸發。
    • 我們注入了 ChangeLogMapper，這是一個使用 MapStruct 產生的對應器，可以輕易地將 DTO (StationChangeEvent) 轉換為用於資料庫儲存的 Entity (StationChangeLog)。
    • 呼叫 logEntry.persist()，Quarkus 的 Panache 套件會自動將這筆異動紀錄儲存到設定好的 MySQL 資料庫的 station_change_logs 資料表中。
  • 同步寫入 Redis 快取：
    • 為了提供快速的即時查詢，我們同時會將這筆異動事件寫入 Redis。
    • 我們使用 Redis 的 List 資料結構，將新的事件從列表的左側 (lpush) 推入。
    • 為了避免 Redis 佔用過多記憶體，我們接著使用 ltrim 指令，只保留最新的 10 筆紀錄。
完整檔案

ChangeQueryService

提供查詢邏輯，並以 Redis 資料優先

  • 先查 Redis：嘗試從 Redis 的 recent_changes 列表中獲取資料。
  • 快取命中 (Cache Hit)：如果 Redis 中有資料，就直接回傳，這會非常快。
  • 快取未命中 (Cache Miss)：如果 Redis 是空的（例如服務剛啟動，還沒有任何異動），則會去查詢 MySQL 資料庫，找出按時間排序最新的 10 筆紀錄，然後回傳。
完整檔案

YouBikeResource

YouBikeResource 中新增定義了 GET /api/youbike/changes/recent 這支 API。

  • 當這支 API 被呼叫時，它會去呼叫 ChangeQueryService 的 getRecentChanges 方法。
